# Steering 管理器

管理 `.kiro/steering/` 下 steering 文件和项目根目录 `AGENTS.md`。新项目从源码扫描生成规则，已有项目结合源码优化现有规则。

核心目标：`AGENTS.md` = 统一主入口 + 专项规范路由器，降低"规则存在但未被读取"概率。

## 快速入口

| 场景 | 执行路径 |
|------|---------|
| 新项目，无 steering 无 AGENTS.md | 第一步（初始化）→ 第一步b（访谈）→ 第三步（整合） |
| 已有 steering 或 AGENTS.md | 第二步（优化）→ 第三步（整合） |
| 用户纠正了 AI 行为，需沉淀为规则 | 第四步（演化） |

## 规则写入标准

### 什么该写

steering 只记录"AI 写代码时大概率遗漏的知识"：
- 跨多文件/层级才能串联的隐含约束，AI 写具体方法时不会主动串联
- 修改 A 必须同步 B，但 A、B 间无代码级引用
- 项目特有模式，与框架默认行为不同，AI 会按默认方式写
- 非标准工具链——训练数据中少见的工具，AI 会按主流工具的方式写

### 强制自检 gate（每条规则写入前必须全部通过）

1. **来源验证**：从代码/配置确认的事实？推断的不写。例外：用户在访谈或纠正中明确陈述的经验视为已验证来源
2. **可见性**：AI 处理相关任务时会自然读到？会的不写
3. **遗漏概率**：不写这条 AI 犯错概率大？概率低不写
4. **文档覆盖**：README、设计文档或其他已有文档已明确描述？已描述不写——重复只增加成本不增加信号
5. **稳定性**：对应的代码/架构短期内会大幅变动？会的暂不写入，标记为待观察
6. **可强制性**：能通过 linter、hook、编译器强制执行？能的不写进 steering，改用工具强制。按约束类型选工具：
   - **流程约束**（构建命令、包管理器）→ shell hook / CI 脚本
   - **源码约束**（代码风格、命名规范）→ linter / AST 静态分析
   - **架构约束**（模块依赖、层级限制）→ 编译器 / ArchUnit 等结构验证
   工具无法覆盖的（如跨模块联动的隐含约束、项目特有的 Why），才写入

### 默认排除（通常无法通过 gate）

- 纯业务流程描述（不涉及跨模块联动约束的）
- 枚举值列表、字段含义对照表
- 单文件可读的实现细节（key 格式、方法签名、类继承关系等）
- 框架通用约定（Spring Boot 默认行为、Express 中间件常规用法等）
- 产品功能概览
- 未经验证的推断或假设
- 测试框架通用用法，但项目特有的测试基础设施约定（自定义启动配置、数据清理策略等）不在此列

以上项目若确实通过了 gate 全部 6 条，附上判定理由后可写入。

## 生成原则

1. steering 文件负责主题内容，`AGENTS.md` 负责路由
2. 触发条件偏宽但有边界：路径和核心关键词命中即读取，同时用排除条件避免误命中
3. 反复漏读或误用代价高的主题 → 提高路由优先级
4. 不预设业务主题，高优先级从源码/目录/历史漏读线索判断
5. 写入前列出变更清单（新增/删除/修改的规则），确认后写入，完成后输出变更摘要
6. steering 文件内所有本地文件路径使用项目根目录的相对路径（如 `.kiro/steering/testing.md`、`src/config/AppConfig.java`），NEVER 用相对于 steering 文件自身的路径（如 `../../`）— AI 以项目根目录为基准读文件

### 变更清单格式

```
### 新增
- [文件名] 规则描述
### 删除
- [文件名] 规则描述（删除原因）
### 修改
- [文件名] 原规则 → 新规则
```

### 变更摘要格式

```
- steering 文件：X 个新增 / Y 个修改 / Z 个删除
- AGENTS.md：行数 N（+/-变化）
- 规则总数：通用 N 条 + 专项 N 条 = 总计 N 条
- 待观察：[列出标记为待观察的规则]
```

## 源码扫描策略（初始化和优化共用）

目标：发现"AI 从局部代码推断不出的隐性知识"。从构建配置（pom.xml/build.gradle/package.json）识别所有模块，按以下 7 类知识逐项排查：

1. **跨模块联动约束**：改 A 必须同步 B，但 A、B 间无代码级引用。方法：读各模块入口和配置，识别跨模块注册点、事件监听、条件装配（Java: Spring 配置类/事件监听；JS/TS: 插件注册/中间件链；Python: signal/hook 注册）
2. **项目特有约定**：与框架默认行为不同，AI 会按默认方式写。方法：读自定义装饰器/注解、中间件/拦截器、AOP 切面，对比框架默认行为找差异
3. **隐含对象边界**：哪些对象不能跨层传递，编译器/类型系统不拦截但违反即为 bug。方法：读各层的类型转换器、Repository/DAO 接口签名、Controller/Handler 入参出参，识别对象流转规则
4. **硬编码注册/路由**：新增功能必须手动注册但无编译期检查。方法：读配置中的手动注册列表、路由表、扫描路径配置（Java: @Bean 注册；JS/TS: 路由数组/插件列表；Python: urlpatterns/app.include_router）
5. **近期架构变更**：旧模式仍可编译但已废弃，AI 会按旧方式写。方法：源码扫描优先——发现同一职责存在新旧两种实现模式时，针对性查相关文件的 git log 确认迁移方向；不做全量 git log 扫描。后续优化时使用增量扫描（见"git 扫描水位线"）
6. **测试基础设施约定**：项目特有的测试启动配置、数据清理机制、最小上下文加载模式，AI 写新测试时会按框架默认方式写。方法：读测试目录下的自定义启动配置/测试工具类（Java: TestApplication/TestConfig；JS/TS: setup files/test helpers；Python: conftest.py/fixtures），识别与框架默认行为的差异。测试规则量较多（≥5 条）时建议独立为 testing 主题文件
7. **高风险模块标注**：识别状态机、并发控制、金额/权限相关逻辑、跨模块协调等高风险模块，在 testing steering 中标注"测试必覆盖模块"：列出模块名、风险点、对应的测试文件

过滤原则：
- 源码中发现的事实，AI 处理相关任务时会自然读到该文件 → 不写入
- "复杂"≠"隐性"，设计文档有明确描述的不写入

### 规则写法示例

```
# ✅ 好规则：正面祈使句 + Why
- 缓存失效放 `@TransactionalEventListener(phase = AFTER_COMMIT)` — 事务回滚后缓存已清，导致脏读
- NEVER 在 `@Transactional` 方法内直接清缓存

# ❌ 坏规则：描述性段落，无法验证
- 缓存管理需要注意事务边界问题，确保在合适的时机进行缓存失效操作
```

## 第一步：初始化（仅新项目）

> 建议新开 session 执行，确保有足够 context 空间容纳源码扫描结果。

1. **扫描项目结构**：读构建配置（pom.xml/build.gradle/package.json），识别模块划分、技术栈、分层边界
2. **提取隐性知识**：按"源码扫描策略"逐项排查，候选规则须通过自检 gate
3. **按主题生成 steering 文件**：每文件一个主题，小写英文短横线命名（如 `api.md`、`data-layer.md`）

steering 文件要求：
- 清晰主题边界 + 具体可验证规则
- 规则用正面祈使句（告诉 AI 做什么，而非不做什么）；需要表达禁止时用 `NEVER` + Why（仅用于硬边界）
- 反直觉规则在同一行内附 Why（用 ` — ` 连接），自解释规则不加
- 一条规则一个意图，复合规则拆成独立条目
- 文件引用代替长代码片段
- 不承担路由职责，路由统一由 `AGENTS.md` 维护

### token 预算

steering 是 AI 注意力预算的一部分，规则越多遵守率越低：
- `AGENTS.md` 通用规范 ≤ 30 条规则（always-load，每条都消耗注意力）
- `AGENTS.md` 总长度 ≤ 500 行；超过 200 行时审视是否有内容应拆到专项规范文件（按需加载比 always 加载更省注意力）
- 单个 steering 文件 ≤ 150 行、≤ 40 条规则
- 单次任务加载的规则总量（AGENTS.md + 命中的 steering 文件）建议 ≤ 80 条
- 信息密度优先级：正面祈使句 > 判定条件 > Why > 示例 > 描述性段落
- 同一约束能用 1 行祈使句表达的，不用 3 行段落描述

### 规则排列顺序

AI 对文件开头和结尾的指令关注度更高（"lost in the middle"效应）：
- 每个 steering 文件：最易违反的规则放开头，最关键的硬约束放结尾
- `AGENTS.md`：启动前检查和通用规范放前部，路由表放中部，回复前置要求放尾部

## 第一步b：结构化访谈（初始化后执行）

源码扫描完成后，不直接输出结果。基于扫描发现主动向用户提问，挖掘代码中看不出来的隐性知识。

### 问题推导原则

问题从扫描结果推导，不是固定模板。例如：扫描发现用了 Spring 但有自定义注解 → 问"还有哪些与 Spring 默认行为不同的约定？"

### 5 类问题方向

1. **反默认约定**：扫描发现了 X 定制，还有其他与框架默认不同的地方吗？
2. **隐含联动**：模块间有没有代码里看不出来的联动约束？（如改 A 必须同步 B，但 A、B 间无代码引用）
3. **架构迁移**：有没有正在进行的迁移，旧模式还能编译但不应再用？
4. **踩坑经验**：团队里有没有新人经常踩的坑？
5. **不成文规定**：有没有口头约定但从没写下来的规则？

### 执行规则

- 每轮提 2-3 个问题，不一次性倾倒全部问题
- 问题必须具体（引用扫描中发现的模块名/技术栈/配置），禁止泛泛而问
- 用户回答后，提炼候选规则，仍需通过自检 gate（gate 第 1 条已放宽：用户口述经验视为已验证来源）
- 用户说"没有了"/"跳过"/"差不多了"时结束访谈，进入第三步
- 访谈轮次上限 3 轮，避免疲劳

## 第二步：优化（已有项目）

> 建议新开 session 执行，确保有足够 context 空间容纳源码扫描结果。

读取所有 steering `.md` 文件，结合源码逐文件优化。

### 执行策略：主 agent 批量读取 + 顺序优化

禁止 sub-agent 并行。原因：每个 sub-agent 独立读源码 → 大量重复读取 + prompt 开销。

步骤：
1. **源码扫描**：按"源码扫描策略"逐项排查，建立全局上下文
2. **顺序优化**：逐文件执行精简（2a）+ 补缺（2b），复用已读源码上下文
3. **跨文件去重**：放到第三步整合时处理

### 2a. 精简

1. 只保留通过自检 gate 的规则
2. **陈旧检测**（逐条检查 drift 信号）：
   - 规则引用的类/方法/配置/路径是否仍存在？
   - 规则提到的命令是否仍在构建配置中？
   - 规则提到的工具/依赖版本是否与 lockfile 一致？
   - 规则描述的模式是否已被近期重构替换？检查方式：读取 AGENTS.md 末尾水位线（见"git 扫描水位线"），仅扫描水位线之后的增量 commit（`git log <watermark>..HEAD`），无水位线时针对规则涉及的文件查 git log 而非全量扫描
   疑似过时的规则确认后删除
3. 删除"默认排除"列表中的内容
4. 一条规则一个意图，复合规则拆成独立条目
5. 精简 token，删非必要空行
6. 反直觉或反复违反的规则在同一行附 Why，自解释规则不加
7. 内容少但命中错误代价高的文件 → 不因"内容少"删除或弱化

### 2b. 补缺

按"源码扫描策略"排查未覆盖的隐性知识。新规则尽量包含稳定主题线索（模块名/目录名/包名/类名/领域词），容易误判的主题把边界写清楚。新规则归属到最匹配的现有 steering 文件；若无匹配文件，暂存待第三步决定是否新建文件或并入 AGENTS.md 通用规范。

如果扫描后候选规则明显偏少（如仅 0-2 条新增），可选择性执行第一步b的访谈流程，向用户提问补充代码中看不出来的隐性知识。

## 高优先级专项规范判定

满足任一条件可提升为高优先级：
- 涉及明确模块边界，误用代价高
- 实现链路长、跨层多、局部代码容易误判
- 用户曾指出 AI 漏读
- 规则不多但漏读就做错方向
- 主题与开发/排查高度耦合，局部阅读难建立完整上下文

高优先级在 `AGENTS.md` 中：靠前位置 + "路径+关键词+场景"组合路由 + 命中时先读取

## 第三步：整合到 AGENTS.md

`AGENTS.md` 放在项目根目录，Kiro / Claude Code / Cursor 等多平台共用。

团队同时使用多个 AI 工具时，用 symlink 保持单一信源，避免多份文件内容漂移：
- `CLAUDE.md` → symlink 到 `AGENTS.md`（Claude Code 原生支持）
- Cursor：在 `.cursor/rules/` 下创建 `.mdc` 文件引用 `AGENTS.md` 内容，或在 Cursor 设置中指定读取 `AGENTS.md`
- 更新只改 `AGENTS.md`，其他工具自动或手动同步

### AGENTS.md 结构

```
# {项目名}
  项目概览          ← 模块职责 + 目录定位（不写架构描述，agent 能从代码推断）
  常用命令          ← 构建/测试/lint 的精确命令，agent 最高频引用
  操作边界          ← Always（自由操作）/ Ask First（需确认）/ Never（绝对禁止）
  启动前检查        ← 任务开始前：检查是否命中专项规范，命中则先读取再继续
  通用规范          ← 跨主题通用规则
  规则冲突优先级    ← 规则冲突时的裁决顺序
  专项规范路由表    ← 每条含：路径、关键词、场景
  回复前置要求      ← 每次回复前：说明已读取了哪些专项规范（可观测性）
  专项规范遗漏补救  ← 用户指出遗漏时的修复流程
```

### 项目概览

- 只写**模块职责定位** + **技术栈精确版本**（从 lockfile/.tool-versions/Dockerfile 提取）
- 不写架构描述——agent 能从代码推断
- 反直觉的设计决策简要说明 WHY，显而易见的不写

### 常用命令

- 从构建配置提取精确的构建/测试/lint 命令（含完整 flags），仅记录非标准命令
- 列出所有非标准工具（如 uv 而非 pip、bun 而非 npm）——AGENTS.md 中提到的工具被 agent 使用频率远高于未提及的
- 命令列表后附"完成定义"：哪些命令全部 exit 0 才算任务完成，具体到命令和退出码

### 操作边界

三层分级，具体到目录或文件模式：
- **Always**：可自由操作（如 src/、tests/）
- **Ask First**：需人工确认（如新增依赖、数据库迁移、CI/CD 配置）
- **Never**：绝对禁止（如 secrets/密钥文件、生成代码、生产环境配置）

从项目结构识别敏感目录（.env、deploy/、vendor/、generated/ 等）。

### 规则冲突优先级

包含规则冲突时的裁决顺序，从高到低排列。从项目实际的错误代价推导——违反代价越高优先级越高。

### 路由表条目格式

每条专项规范包含 4 类信息：
- **路径**：模块目录、包名
- **核心关键词**（5-8 个）：高区分度，出现即命中。选择标准：该词几乎只在此主题下出现，不会在其他模块的日常任务中被提及
- **场景**：开发/排查/review/设计/重构 + 典型动作
- **排除条件**：明确什么情况下不需要读取此规范，减少误命中。例如"纯 Controller 层改动不需读取 service 规范"

关键词精简原则：
- 辅助关键词（区分度低、需结合路径才有意义的）不放路由表，移到对应 steering 文件头部作为自身的主题描述
- 路由表只保留"看到这个词就该读"的核心关键词，避免关键词过多导致 AI 在不相关任务中误判命中

路由表链接路径：使用项目根目录的相对路径（如 `.kiro/steering/service.md`），不使用相对于 AGENTS.md 的路径。AI 读文件时以项目根目录为基准。

### fileMatch 兜底（Kiro 环境）

为每个 steering 文件配置 frontmatter `inclusion: fileMatch` + `fileMatchPattern`，按路由表中的路径生成 glob 模式。

- **fileMatch（机械匹配）**：AI 读到匹配路径的文件时自动加载，100% 精确，作为兜底
- **路由表（AI 判断）**：覆盖 fileMatch 无法捕获的场景（关键词触发、场景触发、用户提问但未打开相关文件）

```yaml
---
inclusion: fileMatch
fileMatchPattern: "{module-name}/**"
---
```

注意：此机制为 Kiro 特有，其他平台路由表仍作为唯一触发机制。

### 更新已有 AGENTS.md（按章节处理）

| 章节 | 处理 |
|------|------|
| 项目概览 | 保留，仅结构变化时追加 |
| 常用命令 | 构建配置变更时同步更新，保持命令精确 |
| 操作边界 | 目录结构或安全策略变更时同步更新 |
| 启动前检查 | 缺失补建，过弱增强 |
| 通用规范 | 从优化后 steering 重新提取，替换 |
| 路由表 | 新文件加链接，删文件移链接，触发条件允许增强不允许无理由缩窄 |
| 回复前置/补救 | 保留并补强 |

反复漏读的专项规范 → 增强关键词/路径/场景 + 提升位置

### 专项规范遗漏补救

`AGENTS.md` 的遗漏补救章节应指导 AI 区分三种违反原因并采取不同修复策略：

1. **未读到（路由未命中）**：触发条件太窄/关键词缺失/路径未覆盖
   → 修复路由表：扩关键词、加路径、放宽场景描述
2. **读到但被冲淡（注意力竞争）**：规则淹没在大量上下文中，AI 处理多个并发请求时跳过
   → 修复规则本身：精简同文件其他规则腾出注意力预算；将该规则移到文件开头或结尾（高关注位置）；考虑措辞是否足够直接（祈使句 vs 描述段落）
3. **读到但自行豁免（AI 风险评估）**：AI 判断当前场景是"例外"而跳过规则
   → 修复规则措辞：明确标注"无例外"（如"UI 改动也需要测试，不存在'只是小改动'的豁免"）；补充排除条件的反面——列出 AI 可能认为是例外但实际不是的场景

诊断方法：用户指出遗漏时，先让 AI 说明是否读到该规则、为何未遵守，再判断属于哪种原因。

### git 扫描水位线

AGENTS.md 末尾维护一个 HTML 注释作为 git 扫描水位线，记录上次扫描处理到的 commit：

```html
<!-- steering-scan: <commit-sha-short> <YYYY-MM-DD> -->
```

- **初始化（第一步）完成时**：写入当前 HEAD 的 short SHA + 日期
- **优化（第二步）完成时**：更新为当前 HEAD 的 short SHA + 日期
- **读取时机**：第二步开始前读取水位线，仅扫描 `git log <watermark>..HEAD --oneline` 的增量 commit，跳过已处理的历史
- **水位线缺失时**：视为首次，源码扫描为主，不做全量 git log；仅在源码中发现新旧模式并存时针对性查相关文件的 git log

### 去重 + 整合规则

去重原则：
- 跨主题通用规则 → `AGENTS.md`
- 主题专属规则 → 对应 steering 文件
- 路由规则 → 只在 `AGENTS.md`

整合操作：
- 多个 `always` 文件合并进 `AGENTS.md`，合并后删原文件
- 被链接的专项规范文件 frontmatter 设为 `inclusion: fileMatch` + `fileMatchPattern`（Kiro 环境），按路由表中的路径生成 glob 模式；非 Kiro 环境设为 `inclusion: manual`
- `AGENTS.md` ≤ 500 行

## 第四步：错误驱动演化

当用户在日常开发中纠正 AI 行为时，主动提议将纠正转化为 steering 规则。此步骤独立于初始化/优化流程，在任何开发会话中均可触发。

### 纠正信号识别

以下表达模式视为纠正信号：
- 明确否定："不要这样做"、"这样不对"、"别用 X"
- 指出正确做法："应该用 X"、"正确做法是..."、"我们这里的惯例是..."
- 重复纠正：用户明确说"这个问题我之前纠正过"、"又犯了"
- 主动要求："加到 steering"、"记住这个"、"写入规则"、"remember this"、"don't do that again"

### 执行流程

1. **识别**：检测到纠正信号
2. **提炼**：将纠正提炼为正面祈使句格式的候选规则（禁止类用 NEVER），附 Why
3. **gate 检查**：候选规则通过自检 gate（gate 第 1 条已放宽：用户纠正视为已验证来源）
4. **提议**：在回复末尾附提示：`💡 这个纠正可以写入 steering，要我提炼成规则吗？`
5. **等待确认**：用户说"好"/"加上"/"写入"时执行；用户忽略或说"不用"时跳过
6. **写入**：
   - 展示候选规则 + 建议归属的 steering 文件
   - 确认后写入对应 steering 文件
   - 写入时在规则末尾附来源注释：`<!-- source: correction YYYY-MM -->`
   - 必要时更新 AGENTS.md 路由表（新关键词/新路径）

### 频率升级

AI 跨 session 无记忆，频率升级依赖以下信号（满足任一即触发）：
- 用户明确说"这个问题我之前纠正过"/"又犯了"/"上次说过"
- 执行第二步（优化）时，发现同一 steering 文件中有多条 `<!-- source: correction -->` 注释的规则
- 触发后：将该规则移到 steering 文件开头或结尾（高关注位置），并在 AGENTS.md 路由表中增强对应条目的关键词

### 规则质量要求

演化产生的规则与扫描产生的规则标准一致：
- 正面祈使句，一条一个意图
- 需要表达禁止时用 NEVER + Why
- 反直觉规则在同一行附 Why

### 与其他步骤的关系

- 演化产生的规则积累到一定量后，可触发第二步（优化）进行整体精简和去重
- 演化不替代定期维护，仍建议每 sprint 执行一次第二步

## 代码变更后 steering 同步

此规则应写入 `AGENTS.md` 通用规范：

变更后检查是否需同步 steering。触发条件（满足任一）：
- 改变 steering 已记录规则的前提/结论
- 引入新跨模块协作约定
- 建立新"不看会写错"模式，编译器不报错、AI 从局部代码无法推断

不需更新：编译器/IDE 能拦住，或从签名/注释/单文件可推断

## 完成标准

优化完成需满足全部条件，满足即停止，不追求措辞完美：
- 每条保留规则通过自检 gate
- 无跨文件重复规则
- 路由表覆盖所有 steering 文件
- `AGENTS.md` ≤ 500 行
- 单次任务加载的规则总量 ≤ 80 条
- `AGENTS.md` 包含"回复前置要求"章节
- 高优先级规则通过反事实检查：假设 AI 没读到这条规则会怎么做？大概率做对 → 降级或删除；大概率做错且不易发现 → 保留；做错但编译/测试能拦住 → 考虑删除

## 注意事项

- 全局 `~/.kiro/steering/rule.md` 不在处理范围
- 只处理项目级 `.kiro/steering/` 和项目源码/配置
- 已有 `AGENTS.md` 按章节更新，不整体覆盖
- 冲突时优先删弱规则或标注适用范围
- 路由表反复漏读某模块时，优先增强路由表触发条件（扩关键词/路径/场景），而非新建模块级文件
- **警惕规则膨胀**：加规则前先问能否用 linter/hook/CI 强制；每次加规则时同时审视是否有可删除或合并的旧规则
- **AI 生成的规则需人工评审**：AI 生成的规则容易与已有文档冗余，增加成本不增加信号
