## 核心理念

源码学习不是线性阅读。代码更像地图而非小说——先定位自己在哪，再决定往哪走。

三个层次的理解（按需深入，不必每次都到最深层）：

| 层次 | 目标 | 方法 | 何时需要 |
|------|------|------|----------|
| **结构理解** | 有哪些模块？它们如何关联？ | 目录结构 + 入口点 + 依赖关系 | 每个项目必须 |
| **行为理解** | 这个模块做什么？输入输出是什么？ | 接口签名 + 测试用例 + 文档 | 大多数场景 |
| **实现理解** | 内部怎么实现的？用了什么算法？ | 逐行阅读 + 调试跟踪 | 仅在需要改代码或深度理解时 |

## 三种阅读策略

根据学习目标选择策略，可混合使用：

### 1. 自顶向下（Top-Down）

从架构到模块到细节。适合初次接触项目。

```
项目 README / 设计文档
    → 模块划分 / 包结构
        → 核心接口定义
            → 关键实现类
```

**优点**：建立全局心智模型，不会迷失在细节中。
**缺点**：前期投入大，可能读了很多暂时用不到的内容。

### 2. 自底向上（Bottom-Up / Debug-Driven）

从一个具体问题或功能点出发，断点调试追踪调用链。

```
设置断点（入口方法 / 关键路径）
    → 单步执行，观察调用栈
        → 记录关键类和方法
            → 反向推导架构
```

**优点**：目标明确，立即获得具体理解。
**缺点**：容易只见树木不见森林。

### 3. 用例驱动（Use-Case Driven）

选一个典型用例，端到端追踪数据流。

```
客户端发起请求
    → 网络层接收
        → 协议解析
            → 业务逻辑处理
                → 存储 / 响应
```

**优点**：理解真实的执行路径，最接近实际运行。
**推荐**：作为主策略，配合自顶向下建立全局视图。

## 使用熟练度确认（首次建档时执行）

在进入源码学习之前，必须先确认用户对该项目的使用经验。

### 确认流程

首次建档时询问：
1. 你用过这个项目吗？用在什么场景？
2. 核心 API / 核心功能你熟悉到什么程度？

### 分流规则

| 使用经验 | 判断标准 | 学习路径 |
|----------|----------|----------|
| **没用过 / 用得少** | 没跑过 demo、不了解核心 API、没在项目中实际使用过 | 先走「阶段 0：使用掌握」，阶段 0 全部完成后才能进入源码学习 |
| **用得很熟** | 在生产项目中使用过、熟悉核心 API 和典型场景 | 跳过阶段 0，直接进入源码学习（阶段 1 起） |

用户自称"用得熟"时，可出 1 个简单的使用场景题验证。通过则跳过，不通过则建议先走阶段 0。

### 阶段 0：使用掌握

目标：能在实际场景中正确使用该项目，理解其能力边界和常见陷阱。

设计依据（业界参考）：
- freeCodeCamp（Liz Johnson）："学 Rails 之前先学 Ruby"——不理解底层语言，碰到非标准问题就卡住
- reintech.io 常见错误总结：把库当黑盒不理解设计模型、忽略文档和 API 变更、不读 migration guide
- Kent C. Dodds："You contribute best to something you use regularly"——先会用再看码

学习内容（按顺序）：

1. **前置知识确认**：该项目依赖的语言特性、核心概念是否已掌握（如学 Netty 前需理解 Java NIO、学 TiKV 前需理解 Rust async）。未掌握则先补前置知识
2. **文档通读 + 能力边界建立**：通读官方文档核心章节（不求记住，建立"它能做什么"的认知地图）。重点关注：核心概念模型、API 设计哲学、版本变更和 migration guide
3. **快速上手**：按官方 Getting Started / Quick Start 跑通最简 demo
4. **核心 API 学习**：逐个学习核心 API，每个 API 配合实际示例动手写（不是只看 demo）
5. **典型场景实践**：2-3 个该项目最常见的使用场景，端到端动手实践
6. **常见陷阱与错误模式**：收集该项目社区中高频踩坑点（GitHub Issues 高频问题、Stack Overflow 常见错误、官方 FAQ/Troubleshooting）。每个坑点讲清：错误写法 → 正确写法 → 为什么
7. **生产级最佳实践**：性能调优参数与基准、错误处理模式（重试/熔断/降级）、安全注意事项、监控与可观测性配置、版本升级策略

完成标准：用户能独立完成一个典型使用场景 + 能说出 3 个以上常见陷阱及规避方法。

**阶段 0 使用 Direct Instruction 教学模式**（SKILL.md §4A）。讲解 API 时完整讲解用法和示例，不中途追问。

## 标准切入流程（六步法）

每个开源项目的学习，按此顺序推进：

### Step 1：源码环境搭建

**目标**：能在本地编译源码、跑通测试、设断点调试。

> 注意：此步骤假设用户已完成阶段 0（或已跳过），已经会使用该项目。这里搭建的是**源码开发环境**，不是使用环境。

- 克隆代码，切到稳定版本（release tag，不用 main/master 最新）
- 按官方文档搭建开发环境
- 编译通过 + 单元测试通过
- 能在 IDE 中设断点并命中

**为什么第一步**：无法运行的代码无法调试，无法调试就只能靠猜。

**完成标准**：项目能编译，测试通过，能在 IDE 中设断点并命中。

### Step 2：全局地图

**目标**：建立项目的心智模型。

- 读 README、CONTRIBUTING、架构文档（如果有）
- 画出模块依赖图（Mermaid）
- 识别核心模块 vs 辅助模块
- 找到入口点（main 方法 / 启动类 / 请求处理入口）
- 了解项目的线程模型 / 事件模型

**关键产出**：一张模块关系图 + 一段话描述"这个项目做什么、怎么做"。

### Step 3：核心数据结构

**目标**：理解项目的"骨架"。

- 找到核心领域模型 / 数据结构（通常在 model/domain/entity 包下）
- 理解关键字段的含义和生命周期
- 画出核心数据结构的关系图

**为什么重要**：数据结构决定了算法和流程。理解了数据结构，代码逻辑就能猜出大半。

### Step 4：主流程走读

**目标**：端到端理解一个最核心的流程。

- 选择项目最核心的一个用例（如：Raft 的选主、Netty 的请求处理、RocketMQ 的消息发送）
- 用 Debug 模式单步跟踪，记录调用链
- 画出时序图（Mermaid sequenceDiagram）
- 标注关键的设计决策点（为什么这样做而不是那样做）

**技巧**：
- 先走 happy path，忽略异常处理和边界情况
- 记录"看不懂的地方"，不要卡住，标记后继续
- 关注数据在各层之间的变换

### Step 5：分支模块深入

**目标**：逐个击破非核心但重要的模块。

- 按里程碑清单顺序，每次深入一个模块
- 每个模块的学习模式：接口 → 核心实现 → 测试用例 → 边界处理
- 与主流程的交互点是重点

### Step 6：设计哲学提炼

**目标**：从"知道怎么做"上升到"理解为什么这样做"。

- 提炼项目的核心设计模式和架构决策
- 对比同类项目的不同选择（如 Raft vs Paxos，Netty vs Mina）
- 总结可复用的设计经验
- 尝试回答："如果让我重新设计，我会怎么做？为什么？"

## 里程碑拆分模板

开源项目的里程碑按模块而非章节拆分：

```
阶段 0：使用掌握（用户已熟练使用时跳过）
  - 里程碑 0.1：前置知识确认 + 补齐
  - 里程碑 0.2：文档通读 + 能力边界建立
  - 里程碑 0.3：快速上手 + demo 跑通
  - 里程碑 0.4：核心 API 学习
  - 里程碑 0.5：典型场景实践
  - 里程碑 0.6：常见陷阱与错误模式
  - 里程碑 0.7：生产级最佳实践

  ⛔ 门禁：阶段 0 全部完成后才能进入阶段 1（源码学习）

阶段 1：源码入门
  - 里程碑 1.1：源码环境搭建（编译通过 + 单元测试通过 + IDE 断点可命中）
  - 里程碑 1.2：全局架构理解 + 模块关系图

阶段 2：核心流程
  - 里程碑 2.1：核心数据结构理解
  - 里程碑 2.2：主流程端到端走读
  - 里程碑 2.3：[项目特定的第二核心流程]

阶段 3：模块深入
  - 里程碑 3.1：[模块 A] 深入
  - 里程碑 3.2：[模块 B] 深入
  - 里程碑 3.N：...

阶段 4：设计提炼
  - 里程碑 4.1：设计模式与架构决策总结
  - 里程碑 4.2：同类项目对比 + 可复用经验
```

**学习流程规范**：源码走读（调试跟踪 + 调用链记录）+ 架构分析（Mermaid 图）

**完成标准**：
- 能画出该模块的核心类图 / 时序图
- 能用自己的话解释该模块的设计决策
- 能在 IDE 中设断点跟踪该模块的关键路径

## 源码教学流程

源码学习默认使用 SKILL.md §4A Direct Instruction 模式。以下是 §4A 在源码场景下的具体实施规则。若用户对某模块已有了解，可切换到 §4B Socratic 模式。

设计依据（业界参考）：
- **Expertise Reversal Effect**（Kalyuga/Sweller）：新手用直接讲解显著优于探究式
- **LACY（FSE '26）**：专家引导的 Code Tour 得分 83%，纯 AI tour 得分 57%。验证放在 tour 之后
- **Simon Willison Linear Walkthrough**：AI 构建结构化代码走读文档，逐文件完整讲解

### 源码场景下的四步流程

| 步骤 | 动作 | 写入时机 |
|------|------|----------|
| 1. 背景确认 | 简要确认用户对今日模块/流程的已知程度 | 收到回答后写入 session |
| 2. 完整讲解 | 设计意图 → 架构图 → 关键源码解读 → 数据流追踪 | 讲解完成后写入 |
| 3. 答疑 | 用户提问，AI 直接解答；用户无问题则进入下一步 | 每轮问答写入 |
| 4. 验证总结 | 1-2 个验证问题 + 本次学习要点总结 | 写入后标记会话完成 |

### 步骤 2：完整讲解（核心步骤）

按以下顺序完整输出，**禁止中途插入提问**：

1. **设计意图**：这个模块要解决什么问题？为什么这样设计？
2. **架构图**：用 Mermaid 画出模块结构 / 调用关系 / 时序图
3. **关键源码解读**：挑 3-5 个关键方法，解释核心逻辑
   - 标注代码中的设计模式（工厂、策略、观察者等）
   - 指出"看这里"的关键行，而非逐行解释
4. **数据流追踪**：一个请求/消息从进入到离开该模块的完整路径

### 步骤 3：答疑

- 讲解完成后询问用户是否有疑问
- 用户提问时**直接解答**，不反问
- 用户无问题则直接进入步骤 4

### 步骤 4：验证总结

验证问题类型（选 1-2 个）：
- **调用链题**："如果客户端发送一条消息，经过哪些类的哪些方法？"
- **设计决策题**："为什么用 [模式 A] 而不是 [模式 B]？"
- **改造题**："如果要给这个模块加 [功能 X]，你会改哪些地方？"
- **故障分析题**："如果 [组件 Y] 挂了，系统会怎么表现？"

避免纯记忆题（"这个类有哪些方法？"）。

验证后输出本次学习要点总结，写入 session 并标记会话完成。

## 源码阅读技巧

### 六个观察维度

阅读任何一段代码时，关注这六个维度：

| 维度 | 关注点 | 示例 |
|------|--------|------|
| **守卫与校验** | 入口处的参数检查、前置条件 | `if (arg == null) throw ...` |
| **默认值** | 缺省行为暗示了设计假设 | `timeout = config.getOrDefault(5000)` |
| **层次边界** | 数据在层与层之间如何转换 | DTO → Domain → Entity |
| **生命周期** | 对象何时创建、何时销毁、谁持有引用 | 连接池、线程池、缓存 |
| **隐藏状态机** | 多个布尔变量组合成的状态 | `isStarted + isReady + isClosed` |
| **错误处理** | 异常是吞掉、转换、还是上抛 | `catch` 块里做了什么 |

### 高效阅读习惯

- **先读测试**：测试是最好的文档，展示了输入输出和边界情况
- **先读接口再读实现**：接口定义契约，实现是细节
- **先走 happy path**：忽略异常分支，理解正常流程
- **用 IDE 的 Go to Definition / Find Usages**：不要线性阅读
- **打断点而非读代码**：运行时的调用栈比静态阅读更直观
- **记录调用链**：用笔记或 Mermaid 图记录，不要只靠脑子

### 代码分类识别

不同类型的代码，阅读重点不同：

| 代码类型 | 特征 | 阅读重点 |
|----------|------|----------|
| **胶水代码** | 连接两个接口，做适配转换 | 两个接口的形状差异，转换是否有损 |
| **接口定义** | public API，模块边界 | 契约完整性，错误处理一致性 |
| **核心实现** | 业务逻辑，算法 | 为什么这样做，与整体的关系 |
| **配置代码** | 初始化、装配、参数 | 自由度有多大，组合爆炸风险 |
| **编排代码** | 协调多个组件的执行顺序 | 事务性、可恢复性、顺序依赖 |

## AI 可利用的辅助手段

AI 导师在开源项目学习中可以发挥独特优势：

### 1. 架构可视化

- 用 Mermaid 画模块依赖图、类图、时序图
- 每个里程碑至少产出一张关键图表
- 图表嵌入 session 文件，形成可回顾的视觉笔记

### 2. 调用链分析

- 利用 AI 的代码理解能力，帮用户梳理复杂调用链
- 输出格式：`A.method1() → B.method2() → C.method3()`
- 标注每一跳的关键参数变换

### 3. 设计模式识别

- 主动识别代码中使用的设计模式并命名
- 解释为什么在此场景下选择该模式
- 对比其他可选模式的优劣

### 4. 对比分析

- 同类项目的设计对比（如 Netty vs Mina，Raft vs Paxos）
- 同一项目不同版本的演进对比
- "如果不这样做会怎样"的反事实分析

### 5. 实践引导

- 引导用户在 IDE 中设断点、单步调试
- 建议用户修改某个参数观察行为变化
- 提出"小实验"：改一行代码，预测结果，验证理解

## 语言速查：项目切入点

任何语言的开源项目，都从以下七个维度找切入点。首次建档时，AI 根据项目实际语言和构建工具自动识别。

| 维度 | 要找什么 | 为什么重要 |
|------|----------|-----------|
| **入口点** | 程序从哪里开始执行 | 理解初始化顺序和启动流程 |
| **依赖管理** | 构建文件 / 包管理器配置 | 模块划分 + 外部依赖一览 |
| **接口/契约** | 公开 API、trait、interface、protocol | 理解模块边界和设计意图 |
| **并发模型** | 线程池 / goroutine / async-await / actor | 理解运行时行为，不同语言差异最大的地方 |
| **扩展机制** | 插件、SPI、中间件、hook | 理解项目如何支持定制 |
| **配置方式** | 配置文件 / 环境变量 / builder 模式 | 理解可调参数和默认行为 |
| **错误处理** | 异常 / Result 类型 / error 返回值 | 跟着错误路径走就是调用链 |

**语言差异提示**（AI 建档时参考，不需要用户记忆）：

- 并发模型差异最大：Java 靠线程池 + Executor，Go 靠 goroutine + channel，Rust 靠 async/await + tokio runtime
- 接口表达不同：Java 用 `interface`，Go 用隐式 interface，Rust 用 `trait`
- 错误处理不同：Java 用异常，Go 用 `if err != nil`，Rust 用 `Result<T, E>` + `?` 操作符
- 扩展机制不同：Java 用 SPI / 注解，Go 用 interface 注入，Rust 用 trait + generic
- 内存管理不同：Java 有 GC，Go 有 GC，Rust 靠所有权系统（学 Rust 项目时必须理解 ownership/borrow）

AI 在首次建档时，根据项目语言自动填写具体的切入点文件路径和惯用模式，不需要在本文件中穷举。

## 领域速查：专项关注点

不同领域的项目，除通用六步流程外，有各自的核心关注点。首次建档时根据项目领域查表。

### 分布式系统（共识、存储、协调）

| 关注点 | 要回答的问题 |
|--------|-------------|
| **一致性模型** | 强一致 / 最终一致 / 因果一致？如何保证？ |
| **故障处理** | 节点宕机、网络分区、消息丢失时怎么办？ |
| **选主/协调** | 谁是 leader？怎么选？脑裂怎么处理？ |
| **数据分片** | 数据怎么分布？路由策略？再平衡？ |

## 典型项目学习路线示例

### 分布式共识：Raft（SOFAJRaft / Hashicorp Raft）
```
环境搭建 → 核心数据结构（Log、Term、State）
→ 选主流程 → 日志复制 → 安全性保证
→ 成员变更 → 快照与日志压缩 → 线性一致读
```

### 消息队列：RocketMQ
```
环境搭建 → 消息模型（Topic、Queue、Consumer Group）
→ 消息发送流程 → 消息存储（CommitLog）→ 消息消费流程
→ 高可用（主从同步）→ 事务消息 → 延迟消息
```

### 网络框架：Netty
```
环境搭建 → 核心抽象（Channel、EventLoop、Pipeline）
→ 启动流程 → 请求处理流程 → 编解码
→ 内存管理（ByteBuf）→ 线程模型 → 性能优化
```

### IM 系统（如 OpenIM / Turms）
```
环境搭建 → 消息模型 + 协议设计
→ 连接管理（长连接 + 心跳）→ 单聊消息收发流程
→ 群聊扩散机制 → 离线消息 + 消息同步
→ 已读回执 + 多端同步 → 推送集成
```

### API 网关（如 APISIX / Spring Cloud Gateway）
```
环境搭建 → 核心抽象（Route、Plugin/Filter、Upstream）
→ 请求处理主流程（匹配 → 过滤 → 转发 → 响应）
→ 路由匹配算法 → 插件/过滤器链机制
→ 限流熔断实现 → 热加载 / 动态配置
→ 可观测性（日志 + 指标 + 链路追踪）
```

### Go/Rust 分布式项目（如 etcd / TiKV）
```
环境搭建 → 项目结构 + 构建系统
→ 并发模型（goroutine+channel / async+tokio）
→ 核心数据结构 → 主流程走读
→ 分布式协议实现 → 存储引擎
→ 测试策略
```

## Gotchas

| 发生了什么 | 规则 |
|-----------|------|
| 还没跑通就开始读源码 | Step 1 强制：先编译通过 + 测试通过 + 断点可命中 |
| 用户没用过项目就直接看源码 | 首次建档必须确认使用熟练度，没用过先走阶段 0 |
| 阶段 0 没完成就进入源码学习 | 门禁规则：阶段 0 全部完成后才能进入阶段 1 |
| 逐行阅读整个文件 | 先读接口签名和测试，按需深入实现 |
| 只讲概念不看实际代码 | 步骤 2 必须包含关键源码解读 + 调用链 |
| 画了架构图但没有时序图 | 核心流程必须有时序图，展示运行时行为 |
| 验证问题是"这个类有哪些方法" | 用调用链题、设计决策题、改造题替代记忆题 |
| 讲解过程中途反问用户 | 步骤 2 必须完整讲解，禁止中途插入提问（§4A Direct Instruction） |
| 答疑阶段用苏格拉底式追问 | 步骤 3 用户提问时直接解答，不反问 |
| 学了主流程就跳到设计提炼 | 分支模块必须按里程碑顺序逐个深入 |
| 忽略了并发模型 | 任何语言的项目都必须理解其并发模型 |
| 用 Java 思维读 Go/Rust 代码 | 先看语言速查的差异提示，理解该语言的并发模型和错误处理惯例 |
| IM 项目只看发消息不看连接管理 | 长连接管理是 IM 的基础，必须先理解 |
| 网关项目跳过过滤器链 | 过滤器/插件链是网关的核心扩展机制 |
